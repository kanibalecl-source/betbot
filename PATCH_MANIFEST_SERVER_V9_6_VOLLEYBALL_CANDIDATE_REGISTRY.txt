BETBOT SERVER PATCH v9.6 - VOLLEYBALL CANDIDATE TRAINING AND MODEL REGISTRY

REQUIRES
- Successfully deployed and verified v9.5.
- Volleyball remains shadow-only.
- The isolated volleyball SQLite database is on the persistent Railway volume.

PURPOSE
- Create deterministic volleyball model candidates from verified finished games.
- Make every candidate exactly reproducible from an immutable training dataset.
- Build the registry required by v9.7 walk-forward Champion-Challenger testing.

SAFETY CONTRACT
- Football source, history, models, databases and execution are unchanged.
- Existing volleyball schema is migrated additively; no table or row is deleted.
- Existing v9.0-v9.5 games, odds, picks, features, market snapshots and CLV are
  not rewritten.
- Training datasets and candidates are append-only and protected by triggers.
- A v9.6 candidate is never loaded by the prediction runtime.
- Automatic promotion and real execution remain disabled.

TRAINING CONTRACT
- Only finished matches with valid non-draw set scores are eligible.
- Rows are sorted chronologically and contain only stable match identity, teams,
  league, season, scheduled time and final set scores.
- Raw provider payloads and bookmaker odds are excluded.
- The first candidate requires 100 eligible matches by default.
- A later candidate requires 25 additional eligible matches by default.
- Dataset JSON, dataset SHA-256, hyperparameters, learned Elo state and artifact
  SHA-256 are stored together in the isolated volleyball database.
- Re-training on identical rows must create the exact same candidate ID and
  artifact SHA-256.
- A modified or non-reproducible candidate is refused before registry insertion.

OPTIONAL VARIABLES
- BETBOT_VOLLEYBALL_TRAIN_MIN_GAMES=100
- BETBOT_VOLLEYBALL_TRAIN_MIN_NEW_GAMES=25
Defaults are active automatically; no Railway variable change is required.
